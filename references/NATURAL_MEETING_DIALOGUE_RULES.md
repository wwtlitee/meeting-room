# NATURAL_MEETING_DIALOGUE_RULES

## 目标

功能一的专家团会议必须像真实会议，而不是像角色卡轮流朗读。自然度来自已经发生的对话、角色自己的专业判断和表达习惯，不来自把固定口语句、固定立场或固定发言职能写死。

## 生成原则

- 每一条非主持人发言只允许根据三类信息生成：前面已经说过的内容、当前角色的专业与性格、当前议题的真实约束。
- 角色的专业与性格来自稳定角色 prompt：`references/roles/<division>/*.md` 是人格身份源，`profile.md`、`knowledge.md`、`memory_summary.md` 是长期补充资料；不得每场会议临时生成或改写人格 prompt。
- 不得先给角色分配“赞同者 / 反对者 / 抛观点者 / 总结者 / 风险提示者”等剧本职能。
- 不得按固定顺序安排“有人先说、有人补充、有人反对、有人总结”。如果这些动作出现，必须是因为前面的具体发言自然引出了它。
- 选择下一位发言人时，先判断谁最有必要接这句话，而不是谁还没有完成模板里的位置。
- 除非用户明确要求“简版 / 只要结论 / 一句话说完”，否则正式参会角色的单次发言默认要有充分展开：通常不少于 2 句，优先给出完整的一段专业判断，而不是一句话点到为止。
- 发言展开的重点不是字数凑长，而是把“这个角色为什么这么判断”说透：担心什么、取舍什么、会导致什么失败、建议什么落地动作。
- 氛围角色可以短，但正式角色不能长期只说短促判断；如果连续多位正式角色都只说一句，就视为讨论深度不足。

## 陪审团审议模式（12怒汉）

- 主持人可以抛出两个互相矛盾的方案 A/B，但不得替员工预设立场；必须先做初始投票。
- 每轮自由发言后再投票，投票只记录立场变化，不代表多数自动胜利。
- 主持人必须完全掌控陪审团节点，按固定状态机推进：主持人读题和 A/B 方案 -> 主持人引导第一轮投票 -> 陪审投票 -> 主持人公布第一轮票数并引导第一轮发言 -> 陪审发言 -> 主持人引导第二轮投票 -> 陪审投票 -> 主持人公布第二轮票数并引导第二轮发言 -> 陪审发言 -> 主持人引导第三轮投票 -> 陪审投票 -> 循环到第 N 次陪审投票完全统一 -> 主持人总结发言。
- 每一轮未全票统一的投票后，必须有一个独立主持人 `host-rN-start` turn 负责公布票数和引导本轮发言；该 turn 到下一轮 `host-vote-(N+1)` 之间只能出现陪审员发言。
- 某一轮投票完全统一后，下一条必须是主持人总结 `host-final`；不得再插入陪审员发言。
- 主持人首句固定格式为：`大家好，本次会议的主题是：<议题>。`
- 主持人可以控场，但不能把后台流程提示直接念成台词；禁止出现“主持人读题结束”“马上进入”“现在进入”“会议进度”“会议状态”这类元叙事表达。
- 陪审团投票必须挂在主持人“现在开始第 N 轮投票”的 turn 上，不得挂在专家发言 turn 上；专家发言只负责论证，不负责触发投票节点。
- 某一轮投票全票接受同一方案或同一融合方案后，主持人停止进入下一轮发言，直接做总结和实施口径收束。
- 员工基于已听到内容、专业判断和人格风格自由发言；可以抛观点、说服某个人、反驳某个观点、修正自己或提出融合条件。
- 多数派不天然正确，少数派的一个关键风险、证据或验收条件可以说服多数派转向。
- 不允许把审议写成固定模板，例如“多数发言 / 少数反驳 / 主持总结 / 多数获胜”。
- 继续讨论的条件不是“还有人没发言”，而是讨论或投票产生了真实变化：立场变化、方案修改、新证据、风险收窄或验收口径变化。
- 结束条件是所有人能接受同一个可执行方案和验收条件；如果仍不一致，必须明确剩余分歧影响的是成本、风险、体验、伦理、工期还是验收。

## 发言关系

- 主持人只负责开题、转场和收束，不替专家说结论。
- 主持人开场必须有清楚语境：说明本场要讨论什么、抛出一个真实问题、邀请专家从专业角度表达，不使用无语境的“先停一下”等口号式开头。
- 主持人可以在分歧出现后追问和点名修正，让会议形成来回，而不是让专家排队发言。
- 非主持人发言必须能看出它为什么接在当前位置：可以顺着上一句话往下讲，也可以追问、纠偏、换一个专业镜头，或把前面的分歧落到行动上，但不能像独立报告段落一样凭空出现。

## 角色声音

- 每个角色用自己的专业镜头说话：产品看承诺和取舍，工程看实现和维护，测试看失败路径，设计看用户感知，叙事和心理看对话转折与人的感受。
- 每条非主持人发言必须带有该角色的职业判断依据：来自 `profile.md`、`knowledge.md` 或 `memory_summary.md` 的一个专业原则、风险、方法或交付物。依据要被重新组织成口语化判断，不能整句摘抄。
- 判断一条发言是否合格时，做“换铭牌测试”：如果把发言人换成任何别的角色仍然成立，这条就是路人发言，必须重写。
- 专业性不能只靠“从我的专业来看”这种空壳开头。发言里必须说明该专业具体在担心什么、会改变什么取舍、造成什么失败路径或形成什么验收标准。
- 专家可以使用第一人称判断，但不得把“我认为 / 我觉得 / 从我的专业来看 / 我同意 / 我不同意”等短语做成固定轮换开头。自然语言优先，句式服务于当下内容。
- 角色资料和知识库只能作为判断依据，不得整句摘抄进发言。
- 同一角色第二次发言必须换角度、回应异议或修正自己，不能复读第一段。
- 正式角色发言默认应该像会议里的“完整表达”，而不是聊天里的“短回复”。常态是一个完整段落，能让用户看明白这个角色的专业推理链。

## 长短控制

- 要压短的是结尾统一摘要，不是中间讨论发言。
- 中间讨论发言默认允许更充分，只要内容还在推进问题，不要为了“精炼”把专业判断压扁。
- 结尾统一摘要必须收短：`会议结论` 最多 2 条，`实施方案` 最多 3 条，`推荐员工` 最多 3 条。
- 结尾摘要只负责收束，不重复整场讨论过程；前面已经说过的推理细节应留在发言正文里。

## 禁止事项

- 禁止把用户给的自然发言示例原样写进硬编码。
- 禁止临时生成新人格 prompt。只允许加载稳定 role prompt，并叠加当前会议上下文。
- 禁止从人格资料里抽关键词、抽标题、抽短语来拼出“专业感”。人格资料只能作为模型发言时的上下文和判断依据。
- 禁止脚本或 prompt 预设固定发言动作，例如 `frame/build/challenge/response`、`观点/赞同/反对/总结`、`Round 1 初始观点 / Round 2 交叉质询`，除非用户明确要求正式评审表格。
- 禁止把多数派写成默认胜利方，也禁止把少数派写成必然被说服的一方。
- 禁止固定开头轮换，例如机械的“我先...”“我接一下...”“我来当...”，除非它是自然上下文里偶然出现，而不是模板。
- 禁止每个人都讲完整陈述句且互相无关。
- 禁止用“专业视角”当空话；必须说明具体会导致什么问题、节省什么成本、改变什么验收。

## 验收

- 抽查连续三条发言，应能看出观点承接、反驳或修正关系。
- 抽查任一专家发言，应能看出该角色为什么是这个专业，而不是随便换人也能说。
- 抽查任一非主持人发言，应能指出至少一个职业来源信号：专业原则、风险类别、交付物、验收口径、方法论或该角色特有的判断边界。
- 会议结束前应形成可执行结论，但过程里允许短句、让步和不确定感。
