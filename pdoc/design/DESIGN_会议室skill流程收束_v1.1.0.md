# DESIGN_会议室skill流程收束_v1.1.0

## 基本信息

- 日期：2026-05-28
- 版本：v1.9.0

## 设计目标

会议室 skill 收束为两个可独立触发的功能：功能一负责“开会出方案”，功能二负责“员工执行子任务”。默认入口是功能一，且用户侧只有一个会议入口：触发后直接进入 text-first discussion 会议，不再询问“普通会议 / 陪审团会议”。陪审/PK/多轮投票只作为显式请求下的内部机制；默认讨论由正式邀请人 + 氛围状态补满 10 席，主进程读取正式人格、可发言氛围状态和实时会议记录，增量写入或会后导入 `meeting-runtime.json`。文字会议是主流程，可视化会议页只作为可选旁观链接，不自动拉起，也不再阻断文字会议。功能一结束后只推荐功能二，并再次让用户选择执行方式。未经会后选择，不自动执行任何子任务。

## 功能边界

| 功能 | 触发意图 | 主要载体 | 输出 |
| :--- | :--- | :--- | :--- |
| 功能一：开会出方案 | 会议室开会、专家团开会、讨论、评审、陪审模式、方案 PK | Codex 文字会议 + 可选 viewer 旁观 | 会议结论、实施方案、推荐员工 |
| 功能二：员工执行子任务 | 开始分工执行、让员工去做、跑子任务 | Codex 工具执行 / 子 agent / 后续工作动画 | 真实产物、执行进度、阻塞项 |

## 触发矩阵

| 用户说法 | 动作 |
| :--- | :--- |
| 会议室 / 开会 / 专家团开会 / 专家们讨论一下 / 你们怎么看 | 直接进入唯一会议入口，默认 discussion text-first live |
| 随便聊 / 聊聊 / 你们怎么看 | 仍进入同一个 discussion live，只作为语气提示，不生成第二条会议分支 |
| 陪审团会议 / 陪审模式 / 12怒汉 / 方案PK / 多轮投票 | 直接进入功能一，陪审团审议作为内部投票机制 |
| 只要结论 / 不要网页 / 简版 | 启动功能一，但跳过可视化链接 |
| 继续会议室 skill / 继续专家团 skill / 更新方向 | 默认进入功能一唯一会议入口，除非用户明确要求执行 |
| 会后选择直接主任务执行 | 启动功能二，但由当前 Codex 主线程直接执行 |
| 会后选择拉起子 agent 分工执行 / 开始分工执行 / 让员工去做 / 跑子任务 | 启动功能二，基于最近一次会议产物拆任务并分配给子 agent 或等效 worker |

## 功能一状态机

1. `entry`：Codex 先输出固定触发句并直接进入唯一会议入口，不出现会议类型二选一。
2. `prepare`：选择真实专业角色，读取稳定角色 prompt 与 persona store。
3. `runtime_init`：默认 `mode: "discussion"`，live 写入 `meeting-runtime.json`；显式陪审/PK/多轮投票才启用 `jury_deliberation`；显式 session 调试才使用 `current-session.json`。
4. `start`：启动或复用 viewer，若服务端返回的会议 `id` 与本次会议一致，则生成手动点击的可视化会议链接。
5. `running`：Codex 先显示会议主题、参会人员和可点击的可视化会议链接（若可用），再在链接下方直接展开文字会议。
6. `meeting`：文字会议实时推进；viewer 只负责旁观发言、投票、铭牌、文字同步、转场和会场动效。
7. `outcome_pending`：会议动画未结束前隐藏实施方案和推荐员工。
8. `done`：viewer 保留全量会议记录，Codex 在会后聊天栏同步结论、实施方案、推荐员工。
9. `execution_select`：Codex 询问 `1，直接主任务执行` / `2，拉起子 agent 分工执行`。

## Codex 反馈格式

触发功能一后第一可见句：

```text
会议室 skill已触发～喵
```

旧的 `请选择会议模式：1，普通会议 / 2，陪审团会议` 提示禁止再出现。

会议开始时：

```text
会议主题：<topic>
参会人员：主持人（会议主持）、<姓名>（<职位>）、...

可视化会议（可选旁观）：
[打开可视化会议](<viewer-url>)（手动点击打开）

以下为文字会议实录：
```

会议结束时：

```text
会议结论：
- <consensus item>

实施方案：
- [<id>] <title>：<deliverable>；负责人：<role>；验收：<acceptance>

推荐员工：
- <name>（<title> / <priority>）：<task>；交付：<deliverable>

请选择执行方式：
1，直接主任务执行
2，拉起子 agent 分工执行
```

子任务执行时：

```text
执行进度：
- [进行中] <worker/task>：<current action>
- [已完成] <worker/task>：<artifact or change>
- [阻塞] <worker/task>：<blocker and required decision>
```

## 浏览器策略

1. 默认输出可点击的 `[打开可视化会议](currentBrowserUrl)`，仅供用户手动打开。
2. 不再自动点击链接，不再自动拉起 viewer。
3. 不再使用侧栏展开、URL 输入、默认浏览器自动兜底等高成本或误判路径。
4. 若 `servedSession.matches = false`，只跳过可视化链接，不得把旧页面当作本次会议。
5. 文字会议始终继续；可视化会议只是旁观能力，不影响聊天内的真实讨论和结论产出。

## 陪审模式规则

- 只有用户明确要求陪审团、12怒汉、方案 PK 或多轮投票时，才使用 `mode: "jury_deliberation"` 和 `deliberation.voteRounds`。
- 氛围状态不是全员静默占位：`nod` / 对对对、`reserve` / 保留意见、`thinking` / 再想想是可发言氛围人格；`zzz` 和 `phone` 主要承担视觉状态。
- 大屏幕 A/B 不只显示“方案A/方案B”；必须显示方案主含义、说明行和当前票数，避免用户忘记两个方案分别代表什么。
- 初始投票所有可见投票者可以伸手。
- 后续投票只有相对上一轮改变 A/B 立场的员工伸手。
- 不改变立场的员工保留 A/B 小标记，不重复伸手。
- 多数派不自动胜出，少数派观点可以通过后续发言改变票数。

## 播放导航规则

- 顶部导航必须提供 `上一条` 和 `下一条`，用于人工回看/推进会议发言。
- `上一条` 在第一条发言时禁用；`下一条` 保持原有推进语义。

## 后续扩展

- “员工正在工作”动画应绑定功能二真实执行状态，不替代实际工作。
- viewer 可增加 worker 状态层：待执行、工作中、阻塞、已完成、产物回填。
- 会议历史可复现 session，但不应触发新的执行。
